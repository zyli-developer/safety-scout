## 0. 本文用途

把"代码长什么样"钉死：目录结构、关键接口签名、依赖注入边界、状态流、测试组织。开始 Phase 1 之前必须有这份文档作为施工蓝图。

## 1. 决策汇总

| 维度 | 结论 | 备注 |
| --- | --- | --- |
| 后端模块布局 | 分层：`routes / services / llm / storage / schemas / tasks` | FastAPI 通行风格；MVP 适用、演进无痛 |
| LLM 抽象 | `Protocol`（不是 ABC） | duck-type 友好，测试桩零 boilerplate |
| 配置 | `pydantic-settings` 读 `.env` | `get_settings()` 用 `@lru_cache` |
| DI | FastAPI `Depends` | 单例工厂集中在 `app/dependencies.py` |
| 后台执行 | `BackgroundTasks` | 应用启动时扫 `status=queued` 的孤儿任务恢复 |
| DB 访问 | stdlib `sqlite3` + per-request connection | 不引 ORM |
| 错误处理 | 自定义异常层级 → 全局 handler 映射 | route 不写 try/except |
| 前端状态管理 | 无全局；URL 参数 + 局部 `useState` | 只有 2 页（首页 / 报告页） |
| 前端 UI 库 | NutUI-React Taro 版 | 报告卡片自撸，其余拼件 |
| 跨端类型共享 | 手写双份，`docs/specs/report-schema.md` 唯一真相 | BE Pydantic + FE TS interface，飘移由"spec 一致性测试"抓 |
| LLM 集成测试 | Record-Replay，层边界录像（不用 vcr.py） | fixture JSON 存 `tests/fixtures/llm/` |
| 图片处理 | **不压缩 / 不缩放**，原图直传 LLM | 上限 15MB；客户端 `sizeType:['original']`。压缩是否引入由 Phase 1 PoC 决定 |

### 2.2 关键接口签名

#### `LLMProvider`（`app/llm/base.py`）

```python
from typing import Protocol
from dataclasses import dataclass

@dataclass
class RawLLMResponse:
    content: str              # 模型原始字符串输出（可能不合法 JSON）
    model: str
    latency_ms: int
    provider_payload: dict    # 调试用全量响应

class LLMProvider(Protocol):
    name: str                 # "doubao" | "deepseek" | "fake"
    model_id: str

    async def analyze(self, image_bytes: bytes, prompt: str) -> RawLLMResponse: ...
```

#### `parser.parse_report`（`app/llm/parser.py`）

```python
async def parse_report(
    raw: str,
    *,
    reprompt: Callable[[str], Awaitable[str]] | None = None,
) -> ReportPayload:
    """4 级容错：
    L1 json.loads
    L2 regex 抽 {...} 再 json.loads
    L3 调 reprompt(原响应) 拿到第二次输出，再走 L1+L2
    L4 抛 LLMParseError → API 返 LLM_PARSE_FAILED
    """
```

`reprompt` 是注入的——parser 不直接依赖具体 provider，单测可独立。

#### `inspection_service.run_inspection`（`app/services/inspection.py`）

```python
async def run_inspection(
    inspection_id: UUID,
    image: bytes,
    provider: LLMProvider,
    repo: InspectionRepo,
) -> None:
    """后台任务体。
    状态机：queued → processing → succeeded | failed
    失败时把 SafetyScoutError 序列化到 error_json；成功时写 report_json。"""
```

返回 `None`：这是 BackgroundTask，结果通过 repo 落库 + 前端 GET 取回。

#### `InspectionRepo`（`app/storage/inspection_repo.py`）

```python
def create(conn: Connection, image_path: str) -> UUID
def get(conn: Connection, inspection_id: UUID) -> InspectionRow | None
def update_processing(conn: Connection, inspection_id: UUID) -> None
def update_succeeded(conn: Connection, inspection_id: UUID, report: ReportPayload, model_meta: ModelMeta) -> None
def update_failed(conn: Connection, inspection_id: UUID, error: ErrorPayload) -> None
def list_orphaned_queued(conn: Connection) -> list[InspectionRow]  # 启动期恢复用
def gc_older_than(conn: Connection, days: int = 7) -> int           # 返回清理条数
```

纯函数 + 传 `Connection`：测试时传一个临时 sqlite 连接即可，不需要 mock 任何东西。

### 2.4 错误处理

```python
# app/errors.py
class SafetyScoutError(Exception):
    code: str          # "INVALID_IMAGE" | "IMAGE_TOO_LARGE" | ...
    http_status: int   # 400 | 429 | 500 | 504
    user_message: str  # 中文友好提示

class InvalidImageError(SafetyScoutError):
    code = "INVALID_IMAGE"; http_status = 400
    user_message = "图片格式不支持，请上传 jpg / png / webp"

class ImageTooLargeError(SafetyScoutError): ...
class RateLimitedError(SafetyScoutError): ...
class LLMTimeoutError(SafetyScoutError): ...
class LLMParseError(SafetyScoutError): ...
```

`app/main.py` 注册全局 handler：

```python
@app.exception_handler(SafetyScoutError)
async def _(req, exc: SafetyScoutError):
    return JSONResponse(
        status_code=exc.http_status,
        content={"error": {"code": exc.code, "message": str(exc), "user_message": exc.user_message}},
    )
```

Route 函数完全不写 `try/except` —— 异常自然冒泡。

### 2.5 数据流

```
POST /api/v1/inspections (multipart image)
  │
  ▼ routes/inspections.py 接 → image_service.validate(size/MIME) → image_service.save → uploads/{uuid}.{ext}
  │
  ▼ inspection_repo.create() → status=queued
  │
  ▼ BackgroundTasks.add_task(inspection_runner.run, inspection_id, image_bytes, provider, repo)
  │
  ▼ 立即 202 + { inspection_id, poll_url, poll_interval_ms, timeout_ms }

[BackgroundTask] inspection_runner.run
  │ repo.update_processing()
  ▼ provider.analyze(image, prompt) → RawLLMResponse
  │ parser.parse_report(raw.content, reprompt=lambda c: provider.analyze(image, REPROMPT_TEMPLATE.format(c))) → ReportPayload
  ▼ repo.update_succeeded(report_payload, model_meta)
  │  or repo.update_failed(error_payload)
```

GET 请求路径很短：路由 → `repo.get()` → 返响应 JSON。

### 2.6 启动期孤儿任务恢复

`app/main.py` 的 `startup` 事件里：

```python
@app.on_event("startup")
async def _recover_orphans():
    with conn_scope() as conn:
        orphans = inspection_repo.list_orphaned_queued(conn)
        for o in orphans:
            inspection_repo.update_failed(conn, o.id, ErrorPayload(
                code="INTERNAL", message="server restarted",
                user_message="服务重启导致任务中断，请重试"
            ))
```

策略：**不重跑**（重跑可能让用户得到不同的报告），直接标失败让前端展示"再试一次"。MVP 阶段足够。

```typescript
const handleClick = async () => {
```

#### 报告页 `/pages/report?id=xxx&pi=2000&to=90000`

```typescript
const { id, pi, to } = Taro.useRouter().params;
const { result, error, isTimedOut } = usePolling<InspectionResponse>({
  fetch: () => getInspection(id),
  intervalMs: Number(pi) || 2000,   // 兜底默认值，应对直接打开 URL 的场景

if (error || isTimedOut)              return <ErrorView .../>
if (!result || result.status !== 'succeeded') return <ProgressIndicator .../>
return (
  <>
    <PlainWarningCard text={result.report.plain_warning} severity={result.report.overall_severity}/>
    <Summary text={result.report.summary}/>
    {sortBySeverity(result.report.hazards).map(h => <HazardCard hazard={h} key={...}/>)}
```

### 3.3 关键 hook：`usePolling`

```typescript
function usePolling<T>(opts: {
  fetch: () => Promise<T>;
  intervalMs: number;
  timeoutMs: number;
  stopWhen: (result: T) => boolean;
}): {
  result: T | null;
  error: Error | null;
  elapsedMs: number;
  isTimedOut: boolean;
}
```

实现要点：
- `setInterval`（不递归 `setTimeout`，在 Taro 里更稳）
- `useEffect` cleanup 时清 interval
- 单测用 `jest.useFakeTimers()` 推时间验证状态机

### 3.4 API 客户端

```typescript
// src/api/client.ts
class ApiError extends Error {
  constructor(public code: string, public userMessage: string) {
    super(userMessage);
  }
}

export async function request<T>(opts: {
  url: string; method: 'GET' | 'POST'; data?: any;
}): Promise<T> {
  const res = await Taro.request({ url: BASE_URL + opts.url, ...opts, timeout: 30000 });
  if (res.statusCode >= 400) {
    const err = res.data?.error;
    throw new ApiError(err?.code ?? 'UNKNOWN', err?.user_message ?? '网络异常，请重试');
  }
  return res.data as T;
}

// src/api/inspections.ts
export function createInspection(image: ChooseMediaResult): Promise<CreateResponse> {
  return new Promise((resolve, reject) => {
    Taro.uploadFile({
      url: BASE_URL + '/api/v1/inspections',
      filePath: image.tempFilePath,
      name: 'image',
      success: (res) => { /* 解析 200/202，封装成 ApiError on 4xx/5xx */ },
      fail: reject,
    });
  });
}

export function getInspection(id: string): Promise<InspectionResponse> {
  return request({ url: `/api/v1/inspections/${id}`, method: 'GET' });
}
```

## 4. 跨端契约

### 4.1 类型同步

**Source of truth：** `docs/specs/report-schema.md`。

- BE：`app/schemas/report.py`（Pydantic）
- FE：`src/types/report.ts`（TS interface）
- 两边各自参照 spec 实现，互不依赖

**飘移检测**：BE `tests/unit/test_schemas.py` 有一个 spec 一致性测试——解析 spec markdown 里的 JSON 代码块，喂 Pydantic 校验。Spec 改了但代码没跟上 → 测试挂。

**变更纪律**（已在 `report-schema.md` 第 3 节写好）：breaking change 必须同一 PR 改三处：spec + BE Pydantic + FE interface。

### 4.2 错误码 → 用户体验映射

`src/utils/errorMessage.ts` 集中定义：

| `error.code` | FE 显示 | `user_message` fallback | 提供"重试" |
| --- | --- | --- | --- |
| `INVALID_IMAGE` | Toast | ✅ | 是（重拍） |
| `IMAGE_TOO_LARGE` | Dialog（含建议） | ✅ | 是 |
| `RATE_LIMITED` | Toast | ✅ | 是（带 60s 倒计时禁按钮） |
| `LLM_TIMEOUT` | 报告页错误视图 | ✅ | 是 |
| `LLM_PARSE_FAILED` | 报告页错误视图 | ✅ | 是 |
| `INTERNAL` | 报告页错误视图 | ✅ | 是 |
| **未知 code** | Toast 显示 `user_message` 或通用兜底 | ✅ | 是 |

`user_message` 是必填字段，BE 新增 code 时旧 FE 不会崩。

### 4.3 ID / 时间 / 编码

| 维度 | 约定 |
| --- | --- |
| ID | uuid v4 字符串（小写、含连字符） |
| 时间 | ISO 8601 UTC（`...Z` 结尾，不带偏移） |
| 字符编码 | UTF-8 |
| 中文 | 简体；不允许 LLM 输出繁体 / 日文汉字 |
| 严重等级 | 字符串枚举 `"high" \| "medium" \| "low"`，不用数字 |

### 4.4 图片传输

| 维度 | 约定 |
| --- | --- |
| 客户端 | `Taro.chooseMedia({ sizeType: ['original'] })` |
| 单张上限 | 15MB |
| 服务端处理 | 不压缩 / 不缩放 |
| Phase 1 PoC 增补 | 5 张样图各跑原图 + 软压 1280 两版对比，决定 v0.2 是否引入压缩 |
| 上传方式 | `multipart/form-data` 单字段 `image` |
| 允许 MIME | `image/jpeg`、`image/jpg`、`image/png`、`image/webp` |

### 4.5 版本与兼容

- 路径前缀 `/api/v1`，v0.x → v1.x 共用
- 报告 schema 的 breaking change 不另立 `/v2`，按 §4.1 同 PR 同步
- 前端不缓存报告（无登录、无历史）

### 5.2 Record-Replay 设计

**层边界录像，不用 vcr.py**。LLMProvider 抽象是天然的录像点。

**Fixture 格式** `tests/fixtures/llm/<case>.json`：

```jsonc
{
  "input": {
    "image_sha256": "a1b2c3...",
    "image_path": "tests/fixtures/images/case_001_scaffold_no_guard.jpg",
    "prompt_version": "v1.2",
    "provider": "doubao"
  },
  "output": {
    "content": "{...}",
    "model": "doubao-vision-1.5-pro",
    "latency_ms": 18432,
    "provider_payload": { /* 调试 */ }
  },
  "captured_at": "2026-05-18T08:23:11Z",
  "captured_by": "scripts/replay_capture.py"
}
```

**录制脚本** `scripts/replay_capture.py`：扫 `tests/fixtures/images/*.jpg` → 调真 LLM → 写 fixture。每改 prompt 跑一次。

**FakeLLMProvider**（`tests/conftest.py`）：按 image SHA-256 查 fixture，找不到抛 `FixtureMissingError`。集成测试通过 FastAPI `dependency_overrides` 注入。
