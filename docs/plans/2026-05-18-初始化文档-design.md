## 3. `docs/specs/prd.md`

**章节**：

1. **产品定位**
   - 一句话：给工地安全员用的 AI 隐患识别工具
   - 主要使用者：工地安全员（首要画像，报告语气与体例为他们调）
   - 次要使用者：项目经理、工人、其他工地相关人员（同入口、无登录、谁拍都能用）
   - 场景：现场巡检时拍可疑场面 → 拿到能直接转给班组的整改建议

2. **MVP（v0.1）范围**
   - **In**：拍照 / 选图 → 上传 → 调豆包 Vision → 渲染结构化报告
   - **Out（v0.1 故意不做）**：登录、历史记录、PDF/图片导出、多工地归档、消息通知、班组分发

3. **核心用户流程**（复用 README 5 步，标注必经/可选）

4. **关键体验约束**
   - 首页只有一个「拍隐患」主按钮，不强制登录、不弹引导
   - 上传过程三段进度反馈：拍照成功 → AI 分析中 → 报告就绪
   - 报告页一屏可见、按严重等级排序、颜色区分
   - 报告中文专业、能直接转给班组（不出现 "I cannot determine..." 这类模型套话）
   - **报告双层语言**：顶部一行口语化警示（任何工地角色都能秒懂），下方面向安全员的专业报告体（含规范条款引用 + 整改建议）

5. **成功标准（MVP 验收）**
   - 微信开发者工具里，对 5 张真实工地图能跑通端到端
   - 5 张图里至少 3 张的报告"看起来像安全员写的"（人工评判）
   - 端到端响应 < 30 秒（上传 + LLM）
   - **每个 phase 单元测试 0 failed / 0 skipped**（详见 §6）

6. **明确不在 v0.1 处理的 5 个 TODO**：登录、OSS、历史归档、导出、班组分发

## 4. `docs/specs/hazards.md` · 隐患分类体系

| 编号 | 类别 | 典型现象 |
| --- | --- | --- |
| H1 | 高处坠落 | 临边洞口无防护、脚手架缺挡脚板、安全带未挂 |
| H2 | 物体打击 | 上方堆料、未戴安全帽、起重区下方有人 |
| H3 | 触电 | 配电箱无门、线路裸露泡水、临电不规范 |
| H4 | 坍塌 | 基坑无支护、模板支撑不稳、堆载过限 |
| H5 | 机械伤害 | 旋转部件无防护罩、维修未停机挂牌 |
| H6 | 火灾 | 易燃堆放、动火无看护、消防器材缺失 |
| H7 | 中毒/窒息 | 受限空间未通风、有毒材料无标识 |
| H8 | 起重伤害 | 吊装区无警戒、吊具变形、信号不清 |
| H9 | 个人防护缺失 | 未戴安全帽 / 反光衣 / 安全带 / 绝缘鞋 / 防护眼镜 / 口罩 / 手套；安全帽未系下颌带 |
| H10 | 其他 / 文明施工 | 通道堵塞、料具乱堆、警示标识缺失 |

每类字段：`code`（H1...）、`name`、`typical_phenomena`（典型现象列表）、`severity_hint`（默认风险等级建议）、`reference_regs`（参考规范条款，可空）。

**关于 H9 的声明**（写进文档）：H9 是巡检视角的工程实用类别，不对应住建部官方事故类型。理由是 LLM 输出更稳、PPE 是巡检高频项、报告渲染需要单独高亮、便于数据聚合。

## 5. `docs/specs/prompt.md` · LLM Prompt 模版

包含：

- **System Prompt**：身份（资深建筑施工安全工程师）、语言（中文）、输出格式约束（严格 JSON）、引用规范的态度（能引则引，不确定时留空别编）
- **User Prompt 拼装规则**：图片 + 一段任务指令；指令里枚举 H1-H10 类别供选择
- **输出要求**：必须包含 `plain_warning` + `summary` + `hazards[]`，每条 hazard 必须含 `category_code` ∈ {H1...H10}
- **容错策略**：
  1. 解析失败 → 先 regex 抽 JSON
  2. 仍失败 → 触发"二次纠正提示"（把原响应给模型让它修）
  3. 再失败 → 返回 `LLM_PARSE_FAILED` 错误码 + 友好中文提示

## 6. `docs/specs/report-schema.md` · 报告 JSON 字段（FE/BE 共享契约）

```json
{
  "inspection_id": "uuid",
  "created_at": "ISO8601",
  "plain_warning": "一句话口语化警示，任何人秒懂",
  "summary": "整体风险等级 + 一句话总结（面向安全员）",
  "overall_severity": "high | medium | low",
  "hazards": [
    {
      "category_code": "H1",
      "category_name": "高处坠落",
      "description": "看到的具体现象（专业用语）",
      "severity": "high | medium | low",
      "regulation": "引用条款，可为空字符串",
      "suggestion": "可执行的整改建议"
    }
  ],
  "model_meta": {
    "provider": "doubao",
    "model": "doubao-vision-xxx",
    "latency_ms": 12345
  }
}
```

`overall_severity` 是后端冗余字段，方便前端列表展示；`model_meta` 给将来 A/B 用，前端可忽略。

## 7. `docs/specs/api.md` · 后端接口契约

### 7.1 `POST /api/v1/inspections`（创建任务）

- Content-Type: `multipart/form-data`
- 字段：`image`（必填）
- 图片限制：≤ 15MB；jpg/jpeg/png/webp；**不压缩 / 不缩放**，原图直接送 LLM（保留细节；客户端 `wx.chooseMedia` 用 `sizeType:['original']`。是否需要降采样由 Phase 1 PoC 验证，见 §8 Phase 1 增补任务）
- 限流：单 IP 10 次/分钟
- 行为：落盘 → 入 SQLite 任务表 → 立即返回任务 ID
- 响应：HTTP 202

```json
{
  "inspection_id": "uuid",
  "status": "queued",
  "created_at": "ISO8601",
  "poll_url": "/api/v1/inspections/{id}",
  "poll_interval_ms": 2000,
  "timeout_ms": 90000
}
```

### 7.2 `GET /api/v1/inspections/{id}`（查询结果）

- 不限流
- 状态机：`queued` → `processing` → `succeeded` | `failed`
- HTTP 200，body 里 status 自表态

```json
{
  "inspection_id": "uuid",
  "status": "queued | processing | succeeded | failed",
  "created_at": "ISO8601",
  "updated_at": "ISO8601",
  "report": { /* 仅 status=succeeded */ },
  "error":  { /* 仅 status=failed */ }
}
```

### 7.3 `GET /api/v1/healthz`（可选）

`{"status":"ok","llm_provider":"doubao"}`

### 7.4 错误码

| code | 含义 | HTTP（仅 POST 入参错时用） |
| --- | --- | --- |
| `INVALID_IMAGE` | 格式错 | 400 |
| `IMAGE_TOO_LARGE` | 超 8MB | 400 |
| `RATE_LIMITED` | 触发 POST 限流 | 429 |
| `LLM_TIMEOUT` | 后端硬超时 90s | GET 响应里 error |
| `LLM_PARSE_FAILED` | JSON 容错耗尽 | GET 响应里 error |
| `INTERNAL` | 其他 | GET 响应里 error / 500 |

错误响应统一格式：

```json
{
  "error": {
    "code": "...",
    "message": "面向开发者的英文短描述",
    "user_message": "面向用户的中文友好提示，前端可直接显示"
  }
}
```

### 7.5 后端实现选型

- 任务表：SQLite，stdlib `sqlite3`
  - `inspections(id TEXT PK, status TEXT, created_at, updated_at, image_path TEXT, report_json TEXT, error_json TEXT)`
- 执行器：FastAPI `BackgroundTasks`（MVP 不引 Celery / Redis / RQ）
- GC：7 天后清理
- 单任务后端硬超时 90s（LLM 调用 60s + 余量）

### 7.6 前端轮询契约

- 收到 POST 响应后等 `poll_interval_ms` 再开始轮
- 每次 GET 后再等 `poll_interval_ms` 继续
- 累计超过 `timeout_ms` 停轮，对用户报"超时，请重试"
- 后端下发节奏，前端不写死

### 7.7 v0.1 不做的接口

登录、历史列表、报告导出、删除、分享。

## 8. `docs/plans/roadmap.md` · 2 周里程碑

**总原则**：每个阶段以"可验证产物"结尾。次序按风险倒序 —— 先打掉最大未知（LLM 输出质量），再写工程外壳。

### 单元测试硬规则（贯穿全局）

> 每个 phase 必须自带单元测试，**不允许跳过**。禁止 `pytest.skip` / `@pytest.mark.skip` / `xfail` / `it.skip` / `describe.skip` / 注释掉测试 / 任何形式绕过。
> 只有所有测试真的通过（0 failed、0 skipped），该 phase 才算完成。
> 如果某个测试难写，要么改设计让它可测，要么显式记录到 issue 讨论 —— 不静默绕过。

### Phase 1 · Prompt PoC（D1–D4）

**目标**：用 5 张真实工地图，验证豆包 Vision 能否输出可用的结构化报告。

- D1：申请豆包 Vision API key；写最小调用脚本 `backend/scripts/poc_doubao.py`
- D2：起草 Prompt v1；跑 5 张图
- D3：迭代 Prompt v2/v3；过程记录到 `docs/specs/prompt-poc-notes.md`
- D3.5（新增）：**压缩对比实验** —— 5 张样图各跑「原图」+「软压到长边 1280」两版，对比 LLM 输出质量。若无明显劣化，记入备忘"v0.2 可加回压缩"；若劣化，记入"永久保持原图直传"
- D4：把通过的 Prompt 固化到 `docs/specs/prompt.md`；JSON 解析 + 容错策略验证

**测试要求**：
- `LLMProvider` 抽象接口契约测试
- JSON 解析 / 容错策略：fixture 一组畸形响应，验证 regex 抽取 + 二次纠正按预期行为
- Report schema 校验：5 张样图的报告 JSON 都符合 `report-schema.md`

**退出门**：
- ✅ 5 张样图人工评判：≥ 3 张"看起来像安全员写的"
- ✅ `pytest backend/` → 0 failed / 0 skipped
- ✅ 决策点：豆包够用 → 继续；不够 → 切 DeepSeek 重跑 D2-D4

### Phase 2 · 后端骨架（D5–D9）

**目标**：把 Phase 1 的 PoC 包成 FastAPI 服务。

- D5：FastAPI 骨架（pyproject.toml、`app/main.py`、`app/config.py`、`.env.example`）；接 `/healthz`
- D6：`llm_provider` 抽象 + Doubao 实现
- D7：`POST /api/v1/inspections` + SQLite 任务表 + BackgroundTasks
- D8：`GET /api/v1/inspections/{id}` + 错误码全集 + rate limit（图片不压缩，见 §7.1 修订）
- D9：pytest 覆盖关键路径（mock LLM 层）；DeepSeek provider 留接口签名

**测试要求**：
- 路由层：POST 入参校验、GET 状态各分支、错误码映射
- LLM provider 抽象：mock provider 验证后端编排
- SQLite 任务存储：CRUD + 7 天 GC
- 限流：单 IP 11 次/分钟应被拒
- 图片校验：>15MB 应拒；非 jpg/png/webp 应拒（不再有压缩逻辑可测）

**退出门**：
- ✅ 3 张 Phase 1 样图通过 HTTP 复现 Phase 1 报告（集成测试，可打真 LLM 或 record-replay）
- ✅ `pytest backend/` → 0 failed / 0 skipped

### Phase 3 · 小程序接入（D10–D14）

**目标**：在微信开发者工具里完成端到端体验。

- D10：Taro 骨架；首页一个大按钮"拍隐患"
- D11：调相机 / 选图 → POST；进度反馈 UI
- D12：轮询逻辑（按后端 `poll_interval_ms` / `timeout_ms`）→ 跳转报告页
- D13：报告页：顶部 `plain_warning` 卡片 + hazards 按 severity 排序染色
- D14：5 张样图端到端走通；修 bug；录演示视频

**测试要求**（Jest / Vitest，按 Taro 模板选）：
- 轮询状态机：mock 后端响应，验证 UI 状态切换
- 报告渲染：给完整 JSON 断言 `plain_warning` 在顶部 + hazards 倒序染色
- 图片选择 / 压缩工具函数：纯函数级 unit test

**退出门**：
- ✅ 5 张样图在小程序里端到端跑通且渲染正确
- ✅ `pnpm test` / `npm test` → 0 failed / 0 skipped

### 贯穿性约束

- 写 phase N 代码前，**先把退出门测试列出来**，走 TDD 节奏
- 跨 phase 回归：phase 2/3 开始时先跑前一 phase 的测试，确认没改坏

### 风险对照表

| 风险 | 缓解 |
| --- | --- |
| 豆包 Vision 输出质量不够 | Phase 1 退出门把关；不过则切 DeepSeek |
| 微信 60s 请求超时 | 轮询模式已规避 |
| 小程序备案 / 域名 | MVP 仅在开发者工具内演示，不部署真机 |
| LLM 费用超预算 | Phase 1 限 5 张样图反复跑；记录 token 用量 |

### 不在 v0.1 的事

## 9. `CONTRIBUTING.md` · 自律手册（最小子集）

### 9.1 分支策略

- `main` 永远可运行
- 功能分支：`feat/<topic>` / `fix/<topic>` / `docs/<topic>`
- 走 PR 自审流程：自己开 PR、自己 merge，靠 diff 视图找"顺手改坏的东西"
- `main` 不直接 push（branch protection）

### 9.2 Commit 规范

Conventional Commits 子集，type 限 7 种：`feat` / `fix` / `docs` / `refactor` / `test` / `chore` / `perf`。

### 9.3 静态检查

**Backend**：`ruff format` + `ruff check` + `mypy --strict`，配置在 `pyproject.toml`
**Frontend**：Taro 模板默认的 eslint + prettier；TypeScript `strict: true`
**Pre-commit hook**（用 `pre-commit` 框架）：lint + format + 类型检查；测试不进 pre-commit（太慢）

### 9.4 测试纪律（重申）

单元测试不允许跳过。详细规则同 §8 单元测试硬规则。

### 9.5 文档同步

代码改动违背 `docs/specs/*` 契约时，**同一个 PR 内**必须改文档；不能"代码先走文档后补"。

### 9.6 显式不做

- 不上 CI（本地 pre-commit 把关够用）
- 不写 ADR（用 `docs/plans/` 日期文档代替）
- 不上 monorepo 工具
- 不写 PR / Issue 模版
